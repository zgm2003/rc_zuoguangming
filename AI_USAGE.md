# AI 使用说明

本项目允许使用 AI，但我没有把 AI 输出当成最终答案。AI 的主要价值是帮助我快速枚举方案、暴露边界条件，然后我根据工程约束做取舍。

## AI 帮助过的地方

1. **需求拆解**
   - AI 帮助把题目拆成接收通知、持久化、异步投递、失败重试、状态查询几个部分。
   - 我保留了这个拆解，因为它符合系统真实边界。

2. **可靠性语义比较**
   - AI 提醒了 at-most-once、at-least-once、exactly-once 的差异。
   - 最终我选择 at-least-once，因为外部 HTTP 调用没有共享事务边界。

3. **失败场景枚举**
   - AI 帮助列出 4xx、5xx、429、timeout、worker crash、长期不可用等情况。
   - 我把这些情况落到 retry policy、attempt 表和 stale processing recovery 中。

4. **文档结构整理**
   - AI 帮助生成 README 的初始结构。
   - 我重新组织为面试可讲的版本：问题理解、系统边界、投递语义、失败策略、取舍与演进。

5. **测试用例设计**
   - AI 帮助把关键行为转化为测试点，例如 retry backoff capped、permanent 4xx 不重试、worker 成功后标记 succeeded。

## AI 建议过但我没有采纳的方案

### 1. 默认使用 Kafka/RabbitMQ

没有采纳。

理由：这个作业的核心是工程判断，不是堆基础设施。第一版流量、并发、运维环境都未知，直接上 MQ 会增加部署和解释成本。数据库队列已经能证明持久化、重试、状态流转这些关键能力。

后续如果吞吐和并发成为真实问题，再演进到 Postgres + MQ。

### 2. 承诺 exactly-once delivery

没有采纳。

理由：外部 HTTP 请求 timeout 后，本服务无法知道供应商是否已经处理。继续重试可能重复，不重试可能丢通知。严格 exactly once 在这里是错误承诺。

我选择 at-least-once，并引入 `idempotency_key` 作为跨系统协作合同。

### 3. 做供应商模板 DSL

没有采纳。

理由：题目说不同供应商 URL/Header/Body 不同，但没有要求本服务理解供应商业务语义。第一版直接接收最终 URL/Header/Body，边界更清楚。模板系统会把本服务变成集成平台，复杂度过早膨胀。

### 4. 做管理后台

没有采纳。

理由：管理后台对展示有帮助，但不是可靠投递的核心路径。第一版用 API 查询状态和 attempt 记录即可。

### 5. 微服务拆分

没有采纳。

理由：当前系统只有一个职责：可靠投递通知。API、repository、worker 可以在一个服务里保持清晰模块边界。拆成多个服务只会增加部署和一致性问题。

## 我自己做出的关键决策

1. **先持久化再投递**
   - 原因：避免业务系统被外部供应商延迟和故障拖住。

2. **返回 202 Accepted**
   - 原因：API 只表示通知任务已被可靠接受，不表示外部供应商已处理。

3. **选择 at-least-once**
   - 原因：这是 HTTP 外部投递下诚实且可实现的语义。

4. **支持 idempotency_key**
   - 原因：至少一次投递天然可能重复，需要把业务事件唯一性显式传递给接收方。

5. **使用数据库表作为第一版队列**
   - 原因：降低部署复杂度，同时保留持久化、状态查询、重试和恢复能力。

6. **记录 notification_attempts**
   - 原因：可靠性不能只看最终状态，排障时必须知道每次投递发生了什么。

7. **恢复 stale processing 任务**
   - 原因：worker 可能在 claim 任务后崩溃。没有恢复机制，任务会永久卡住。

8. **大多数 4xx 不重试**
   - 原因：多数 4xx 是请求格式或鉴权问题，重试不能解决，只会制造噪音。

## 我对 AI 输出的修正

AI 很容易给出“大而全”的答案：MQ、DLQ、分布式锁、K8s、监控、管理后台、模板系统一次性全上。这些东西不是错，但第一版全上就是过度设计。

我的修正原则是：

> 只实现可靠投递的闭环，把未来演进写清楚，但不把未来复杂度塞进第一版。
